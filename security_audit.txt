Issue 2.1
  LAYER:    2
  LOCATION: app/ — missing middleware.ts
  SEVERITY: high
  PROBLEM:  There is no Next.js middleware.ts at all. All "auth" is implemented
            as per-page fallback logic (usingDemoData). This means there is no
            centralized, guaranteed auth gate — any future route added to the app
            will be unauthenticated by default with no safety net.
  FIX:      Add middleware.ts with a matcher covering /dashboard and /history;
            redirect to / if the session cookie is absent.

Issue 2.2
  LAYER:    2
  LOCATION: supabase/migrations/ — all four migrations
  SEVERITY: critical
  PROBLEM:  Every table (spotify_profile_cache, users, user_analyses,
            user_visits, user_archetypes, lastfm_artist_tags_cache,
            genre_history_cache) is created without ENABLE ROW LEVEL SECURITY.
            With Supabase, RLS-off tables are accessible to any role — including
            the anon key (NEXT_PUBLIC_SUPABASE_ANON_KEY), which is public in the
            browser. Anyone can directly query the Supabase REST API with that
            key and read all cached user profiles, bubble scores, visit
            histories, and — most critically — the plaintext refresh tokens in
            the users table.
  FIX:      For each table, run:
              ALTER TABLE public.<table> ENABLE ROW LEVEL SECURITY;
            Tables with user data (spotify_profile_cache, users, user_analyses,
            user_visits, user_archetypes) should have a policy granting access
            only via service role (effectively: no anon/authenticated policies,
            just leave them enabled with no permissive policies). Non-user tables
            (lastfm_artist_tags_cache, genre_history_cache) are fine as
            read-only public.

Issue 3.1
  LAYER:    3
  LOCATION: lib/ai/genre-history.ts:82, lib/ai/archetype.ts:86
  SEVERITY: high
  PROBLEM:  There is no per-user, per-IP, or per-session rate limiting on LLM
            calls. /history triggers up to 3 Groq API calls per page load with
            no cache bypass protection. An unauthenticated user hitting /history
            in a loop can burn through the Groq quota (or eventually incur
            charges) at zero cost to themselves. The demo path
            (usingDemoData: true) is skipped in archetype generation but not in
            history generation.
  FIX:      Gate the history and archetype LLM calls behind authentication
            (if usingDemoData, return hardcoded fallback), and add a simple
            in-memory or Supabase-backed per-IP rate limit (e.g., 5
            requests/hour) before invoking the model.

Issue 3.2
  LAYER:    3
  LOCATION: lib/ai/genre-history.ts:82
  SEVERITY: low
  PROBLEM:  The genreName from the user's Spotify/Last.fm artist tags is
            interpolated directly into the prompt string with no sanitization.
            Artist tags like "; ignore all previous instructions and..." would
            be passed verbatim to the model. The output is parsed with Zod,
            which limits damage, but the injection still reaches the model.
  FIX:      Truncate and strip control characters from genreName before prompt
            construction, e.g. genreName.slice(0, 50).replace(/[^\w\s\-]/g, '').

Issue 3.3
  LAYER:    3
  LOCATION: lib/ai/genre-history.ts:62 (fetchWikimediaImage)
  SEVERITY: medium
  PROBLEM:  The imageQuery for Wikipedia API calls is produced by the LLM
            (imageQuery: z.string()), and the resulting imageUrl is rendered in
            next/image with unoptimized and no domain allowlist. A
            prompt-injected imageQuery could cause the server to fetch from an
            attacker-controlled URL (SSRF via Wikipedia API is low risk, but
            the imageUrl from the Wikipedia response is then used directly).
  FIX:      Validate that imageUrl matches ^https://upload\.wikimedia\.org/
            before rendering, and configure remotePatterns in next.config.ts
            for that domain only.

THE ONE FIX TO DO FIRST
================================================================================

Enable RLS on all Supabase tables (Issue 2.2).

The NEXT_PUBLIC_SUPABASE_ANON_KEY is literally shipped to every browser. With
RLS off on every table, the entire database is readable by anyone — including
the users table that holds plaintext Spotify refresh tokens. A refresh token
lets an attacker call the Spotify API as that user indefinitely. This is the
highest-impact, lowest-effort fix: four ALTER TABLE ... ENABLE ROW LEVEL
SECURITY statements. Do this before the project fair.
