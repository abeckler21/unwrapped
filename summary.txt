V1 Work Summary

For V1 of Unwrapped, we built the project from a planning-only repository into a working Next.js application with the core "Mirror" experience in place. We started by scaffolding a Next.js App Router app with TypeScript and Tailwind, added validation scripts, installed the main V1 dependencies, and set up a repeatable development workflow with linting, type-checking, and production builds.

We implemented the first version of the Filter Bubble Score. The scoring system now analyzes genre concentration, artist repetition, popularity skew, and temporal consistency across Spotify listening windows. Each score is typed, explainable, and produces a tier such as "Wide Open" or "Narrowing," along with an estimated organic versus algorithmic listening ratio.

We added Spotify OAuth foundations using the Authorization Code with PKCE flow. The app now includes login, callback, logout, session cookies, token exchange, refresh-token handling, and graceful fallback behavior when a session is missing or invalid. We also added Spotify profile fetching for user info, top artists, top tracks, playlists, recently played tracks, and saved-track counts.

We wired Supabase into the V1 data layer. The app now has an admin Supabase helper, profile cache reads and writes, an initial schema migration, and a 24-hour cache path so repeat visits do not always hit Spotify. We also added a cache completeness guard so older cached profiles that are missing required time ranges are ignored and refreshed.

A major part of the work was building the dashboard experience. The dashboard now supports the three V1 listening windows: 4 weeks, 6 months, and all time. It shows the Filter Bubble Score as the hero element, score breakdowns, estimated organic and algorithmic listening, average track length, genre footprint, top artists, top tracks, and macro trend callouts. We also redesigned the dashboard layout into a more polished, demo-ready interface with a centered time-range control, score hero, compact sub-score sidebar, genre and artist section, track list, macro trend grid, and source/honesty footer.

We added seeded demo data and seeded macro trend data so the app can still be demonstrated before live credentials or real cached Spotify data are available. The homepage, setup page, dashboard, and share preview route were updated to reflect the Unwrapped product framing instead of the default scaffold.

We handled several real-world Spotify edge cases. During testing, Spotify returned thinner-than-expected artist metadata for this app/user path, and a follow-up artist enrichment endpoint returned 403 Forbidden. To keep the dashboard honest, we removed misleading fallback values, stopped showing unreliable artist follower/popularity rows, and added optional Last.fm enrichment with Supabase caching so genre analysis can recover when Spotify withholds genre tags.

We also fixed chart rendering issues. The original Recharts-based genre donut was fragile because of client-side container measurement problems, so we replaced it with a deterministic SVG donut that server-renders reliably. Other macro charts use a shared measured chart frame and element-size hook to avoid width warnings while staying responsive.

Finally, we kept the project log updated throughout the work and verified the latest V1 slice with npm run type-check, npm run lint, and npm run build. The latest committed version includes the verified URL-driven listening-window switching, the redesigned dashboard, the cache completeness guard, and the fixed genre footprint chart.

V2 Work Summary

V2 moved Unwrapped from a local/demo-oriented V1 dashboard into a production-tested product at https://unwrapped-one.vercel.app. The deployment work included linking the repo to Vercel, pushing production and preview environment variables, and adding the production Spotify callback URL so OAuth could be tested in the same environment users would actually hit. The app also received an AI layer through the Vercel AI SDK, using Groq in production and Ollama-compatible local models in development.

The main new dashboard feature is the listener archetype system. V2 added rule-based archetype assignment from the user's medium-term Filter Bubble Score, then uses an LLM to generate a sharper, personalized 2-3 sentence reading based on the user's score, top artists, top genres, algorithmic exposure, and temporal consistency. Archetypes are cached in Supabase for seven days. V2 also added temporal bubble tracking: each real user visit records a daily bubble score, the dashboard shows a score-over-time chart, and returning users get an AI-generated trend narrative.

V2 also introduced the /history page, which turns the user's top genres into short music-history cards. For each genre, the app generates origin context, key pioneering figures, Wikimedia-sourced portraits when available, and a horizontal lineage rail showing how the genre developed over time. The history system is cached in Supabase for 30 days and was hardened after production debugging: structured-output issues with Groq were replaced with plain JSON generation and parsing, schema constraints were relaxed, and partial history failures now show a visible notice instead of silently dropping cards.

After review feedback, V2 received a polish and reliability pass. The nav is now auth-aware, dashboard range switching is client-side and no longer causes a full server round-trip, loading skeletons were added for dashboard and history, share links can be copied directly, and confusing or unfinished UI copy was removed. The archetype card now clearly states it is based on the 6-month profile, sub-scores explain what high values mean, /history got cleaner navigation and monogram image fallbacks, and the lineage rail now truncates labels based on actual available spacing. All of this was verified with linting, type-checking, and production builds.

V3 Work Summary

V3 was a security hardening pass conducted as a full seven-layer audit of the production application. The audit covered secrets and environment variables, auth surfaces and database access control, AI input validation and rate limiting, dev/preview/prod environment separation, CI gates, monitoring, and supply chain. Fourteen issues were identified across all severity levels, and the critical and high/medium issues were resolved in this version.

The most critical fix was enabling Row Level Security on all Supabase tables. Every table in the schema had been created without RLS, meaning the publicly-exposed anon key (NEXT_PUBLIC_SUPABASE_ANON_KEY) could have been used to query any table directly via the Supabase REST API. This included the users table, which stores Spotify refresh tokens in plaintext. RLS was enabled on all seven tables, with permissive read-only policies added only for the two shared non-user cache tables (lastfm_artist_tags_cache and genre_history_cache).

The /history page was previously making up to three Groq API calls per page load with no authentication requirement. An unauthenticated visitor could loop the page and exhaust the Groq quota. The fix gates all LLM calls behind authentication: unauthenticated visitors now see a prompt to log in rather than triggering model calls, matching the behavior already in place for the archetype system on the dashboard.

GROQ_API_KEY and OLLAMA_BASE_URL were being read directly from process.env in lib/ai/client.ts, bypassing the centralized env schema in lib/env.ts. Both variables were added to the schema and client.ts was updated to consume them through the validated env object.

A middleware.ts file was added. The app intentionally shows demo data to unauthenticated visitors on page routes, so the middleware does not redirect from /dashboard or /history. Instead it establishes the pattern for protecting future API routes: any route under /api/user/ requires a valid session cookie and returns 401 otherwise.

The Wikimedia image URL returned by the Wikipedia API (based on an LLM-generated query string) was being rendered without domain validation. A check was added to reject any image URL that does not begin with https://upload.wikimedia.org/, and the domain was added to next.config.ts as the only allowed remote image pattern.

The session cookie Secure flag was previously tied to NODE_ENV === "production", which left preview deployments on Vercel (which run over HTTPS) with non-Secure cookies. The check was updated to also treat any deployment with a VERCEL_ENV variable set as production-equivalent.

An agent deny list was added to CLAUDE.md. The list enumerates actions that require explicit human confirmation before any AI agent performs them, including reading or writing environment files, running production deployments, modifying Supabase migrations, and outputting secret values.

A security_audit.txt file was committed to the repository with the full fourteen-issue audit report, severity ratings, and recommended fixes for all remaining lower-priority items.

After the security audit, V3 completed the remaining seven lower-priority items, bringing all fourteen issues to resolved status.

A GitHub Actions CI pipeline was added at .github/workflows/ci.yml. The workflow runs on every push and pull request to main. It runs npm ci, then lint, type-check, and build in sequence. A separate secret-scanning job runs TruffleHog against the git history on every push to catch any accidental credential leaks before they are merged.

Sentry was integrated for server-side error visibility. The @sentry/nextjs package was installed and three config files were created: sentry.server.config.ts, sentry.edge.config.ts, and sentry.client.config.ts. The Next.js config was wrapped with withSentryConfig. Sentry is a no-op unless NEXT_PUBLIC_SENTRY_DSN is set as a Vercel environment variable — no real DSN is hardcoded in the repo. This gives the app production-grade error alerting for API route failures, LLM call errors, and Supabase write failures once the DSN is configured.

LLM call logging was added to both lib/ai/archetype.ts and lib/ai/genre-history.ts. After each generateText call, a structured JSON object is written to stdout: event, model ID, input and output token counts, and a genre or userId identifier. Vercel captures stdout as structured logs, so usage spikes are now visible in the Vercel dashboard without any additional logging infrastructure.

The genreName parameter in generateGenreHistory was sanitized before prompt interpolation. The value is now truncated to 50 characters and stripped of anything outside word characters, spaces, and hyphens before being embedded in the LLM prompt. This limits the reach of any prompt-injection attempt via a malformed Last.fm artist tag.

The refresh_token column was dropped from the users table. A search across the entire codebase confirmed that no application code writes to this column — refresh tokens are stored exclusively in httpOnly session cookies. Migration 006_drop_refresh_token.sql removes the column, eliminating the plaintext token-at-rest risk flagged in Issue 1.2. The migration must be run manually in the Supabase SQL editor.

Issue 4.1 (Vercel environment variable scoping) cannot be automated — it requires a manual audit of the Vercel dashboard. Instructions for confirming that SUPABASE_SERVICE_ROLE_KEY, SPOTIFY_CLIENT_SECRET, and GROQ_API_KEY are scoped to Production only are provided as a follow-up action item for the developer.

V3 also added the first product-version feature: Comparative Mode. An authenticated user visits /compare, which creates (or reuses) a 7-day session token stored in a new compare_sessions Supabase table and displays a shareable link. When a second user opens the link and logs in, both profiles are loaded from the Spotify profile cache and a full side-by-side comparison is rendered at /compare/[sessionId].

The comparison shows bubble scores for both users as large numbers with a proportional divider bar, a plain-English verdict naming who is more algorithmically captured and by how much, sub-score breakdown bars for all four components (genre concentration, artist repetition, popularity skew, taste consistency) with the higher bar highlighted per row, shared artists from both users' top-20 medium-term profiles shown as accent-colored pills, shared genres and a panel of genres unique to each listener, and a divergence score (0–100) quantifying overall profile distance. The page handles edge cases: expired links, unauthenticated visitors, the initiator viewing their own link, and the initiator's profile not yet being in cache. A loading skeleton is shown while the async data loads. A Compare link was added to the nav for authenticated users.

V3 also added Escape the Bubble at /escape. Authenticated users see a list of artists that Spotify's algorithm has been filtering out of their recommendations. The pipeline identifies genre gaps (genres absent from the user's top 8), searches Spotify for artists tagged with those genres, filters to popularity 20–55 (visible but not algorithmically prominent), and scores each candidate by an anti-algorithm score combining inverse popularity and genre distance from the user's listening centroid. Top two artists per gap genre are shown with artist image, genres, popularity bar, and an anti-algorithm score badge. Each artist card links to their Spotify page and shows the top track from their catalog. A Save All as Playlist button calls a new POST /api/user/escape/playlist route that creates a public Spotify playlist in the user's library containing one track per recommended artist. Results are cached for 24 hours in a new escape_cache Supabase table. An Escape link was added to the nav for authenticated users.

V3 added Playlist Autopsy at /playlist and /playlist/[playlistId]. Any public Spotify playlist can be analyzed by pasting its URL or ID. The page fetches up to 200 tracks and all contributing artist genre tags, then computes an Algorithm Score out of 100 using five signals: playlist name matching known algorithmic patterns such as Discover Weekly or Daily Mix (max 30 points), Spotify ownership of the playlist (max 20), mean popularity above 45 (max 25), fraction of tracks released in the last two years (max 15), and genre concentration measured by HHI (max 10). The result is classified as Algorithmic (≥55), Human-curated (≤20), or Mixed. The page also shows an era distribution bar chart grouping tracks by decade, a top-genre breakdown with proportional bars, and a full track list with album art, release year, popularity bar, and duration. Results are cached for 6 hours in a new playlist_cache Supabase table. A Playlist link was added to the nav for authenticated users.

V3 added the Artist Perspective Dashboard at /artist and /artist/[id]. The search landing page queries a new GET /api/user/artist/search route that proxies to Spotify's search endpoint and returns up to five matching artists with image, genres, and popularity. Selecting an artist loads a full career analysis at /artist/[id]. The analysis fetches the artist profile, all albums and singles (up to 50, deduplicated by name), and full album details in batches of 20. For each album it computes an Algorithm Optimization Index (AOI): shortScore represents how close the average track duration is to 3 minutes (1.0 at 3 min, 0.0 at 5 min), and collabRate is the fraction of tracks featuring more than one artist. AOI equals round((shortScore × 0.6 + collabRate × 0.4) × 100). The artist page renders a song length trend line chart comparing each album's average track length against a 3:30 industry reference line, a bar chart of AOI per album with brighter bars at higher scores, a year-by-year popularity trajectory as proportional bars, and a full discography table with cover art, track count, average duration, and AOI. Genre drift is noted when the artist has releases spanning four or more years and their tagged genres changed. Results are cached for 24 hours in a new artist_cache Supabase table. Top artists in the dashboard now link to their /artist/[id] page. An Artist link was added to the nav for authenticated users.

V3 added Rich Shareable Cards. Three focused share pages were added as sub-routes of the existing /share/[userId] overview: /share/[userId]/bubble shows the Filter Bubble Score with sub-score breakdown bars and organic/algorithmic/duration stats; /share/[userId]/archetype shows the LLM-generated listener archetype name and prose with top genre pills as context; /share/[userId]/fingerprint shows the full genre distribution with proportional bars, HHI context, and a concentration note. Each sub-page has a generateMetadata export that sets a descriptive title and description for link previews.

Each of the four share routes (the overview plus the three sub-pages) has a colocated opengraph-image.tsx file that generates a 1200×630 dynamic OG image using next/og ImageResponse (Satori). The images use Inter 400/700 loaded from Google Fonts CDN via a shared lib/og/font.ts helper that fetches the Google Fonts CSS and parses out the woff2 URL. The bubble card shows the score as a large number with tier label and four breakdown bars; the archetype card shows the archetype name prominently with a prose excerpt and genre pills; the fingerprint card shows the full genre list as proportional horizontal bars; the overview card shows the score, tier, archetype name, and top 3 genres. All cards share the same dark #09080e background, orange accent, and UNWRAPPED brand text.

Per-insight share buttons were added to the dashboard. InsightShareButton is a small copy-to-clipboard pill that builds the full URL from window.location.origin, copies it to clipboard using navigator.clipboard with a textarea fallback, and flashes a checkmark for 2.2 seconds. Buttons appear on the archetype panel, the score hero description, and the genre footprint section header. All three are suppressed when the user is viewing demo data.
